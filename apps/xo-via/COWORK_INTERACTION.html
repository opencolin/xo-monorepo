📋Endpoint catalog

Chat — xo-cowork-api

POST/api/chat/prompt
Prepare a turn. Fast, returns {stream_id, session_id} before LLM starts.
Phone-OS route: POST /api/chat
SSE/api/chat/stream/{stream_id}
Long-lived SSE. Events: session-created text-delta heartbeat done agent-error client-action {kind,...}. 600s reconnect window.
Phone-OS route: GET /api/chat/stream/[id] (piped)
POST/api/chat/abort
Cancel in-flight stream. Called when user taps Stop button.
Phone-OS route: POST /api/chat/abort

Files — xo-cowork-api

POST/api/files/list-directory
List dir. Body: {path}. Returns {path, parent, dirs[], files[]}. Clamped to $HOME. Never recurse — call per subdir.
Phone-OS route: GET /api/cowork/projects/[id]/files
POST/api/files/content
Read text file. Body: {path}. Returns {path, content}. For binary use /api/files/content-binary.
Phone-OS route: GET /api/cowork/projects/[id]/files/read
POST/api/files/mkdir scaffold:true
Create dir and optionally scaffold xo-project tree from project_template/. Used as workaround for create-project (step 1 of 2).
Phone-OS route: POST /api/cowork/projects (workaround step 1)

Config + health — xo-cowork-api

GET/api/config/workspace
Returns {roots: {default: "/home/user/xo-projects"}, default: "openclaw"}. Call this first on boot. Never hardcode the projects root.
Phone-OS route: GET /api/cowork/workspace
GET/health
Liveness probe. Returns {status: "ok"}. Used for the offline detector.
Phone-OS route: GET /api/cowork/health

User + workspaces — xo-swarm-api

GET/api/users/me blocker 3
Current user profile. Expects Clerk JWT in Authorization. Returns id, username, email, plan.
Phone-OS route: GET /api/swarm/me
GET/api/workspaces blocker 4
List user's Coworker workspaces with their cowork-api URLs and status.
Phone-OS route: GET /api/swarm/workspaces
POST/api/workspaces blocker 4
Provision new Coworker workspace. Returns id + cowork_url once ready. "Create Your Own" CTA triggers this.
Phone-OS route: POST /api/swarm/workspaces
GET/api/workspaces/{id}/connection blocker 5
Returns live cowork-api URL for a workspace. Next.js uses this URL for all cowork calls for the signed-in user.
Phone-OS route: GET /api/swarm/workspaces/[id]/connect

❓Open questions

  • X-XO-Username header. Does xo-cowork-api currently scope data per this header, or is that new work?
  • Operator fallback. When signed out, should Next.js send X-XO-Username: __operator__ or omit the header entirely?
  • Workspace URL per user. After sign-in, is the user's cowork-api URL always :5002 (same host), or a different URL per provisioned workspace (e.g., a Coder or Vercel Sandbox URL)?
  • client-action events. How does the cowork-api agent infrastructure emit client-action events — tool call, structured output, or special token? This determines whether action-driven navigation needs backend work.
  • swarm-api auth scheme. Does swarm-api accept Clerk JWT directly in Authorization, or does it need a specific swarm-issued token?
  • Single session per user. Does cowork-api auto-resolve the user's single session from X-XO-Username on chat calls, or does the phone OS need to discover + pass a session_id first?
  • Quip source. Is there (or will there be) a /api/quip endpoint on cowork-api or swarm-api for Thought of the day, or does the phone OS keep a local data/quips.ts?
  • Workspace provisioning time. How long does POST /api/workspaces take? If it's async, how does the phone OS know when the workspace is ready (polling? webhook? SSE)?
  • Blockers timeline. Blockers 1+2 on cowork-api, and blockers 3–5 on swarm-api — which land first, and in what order?
