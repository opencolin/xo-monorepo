Pivot, 2026-05-20. The TypeScript agent backend (Anthropic SDK, /api/agent/*, lib/agent/*) is deleted. v.Next ships the phone OS UI shell only: lockscreen, pager (3 modes + sub-pages), chat dock button + expand animation, widgets that work locally, consolidated overlay. cowork-api integration is a v.Next+ feature drop (Sections B, G, H, J marked deferred). This means the v.Next build plan reduces to 5 sections: A, C, D, E, I. Each can be handed off to a fresh agent.

1.New architecture, thin client over cowork-api

The phone OS no longer hosts an agent backend. xo-cowork-api owns chat, sessions, files, agents, usage. The Next.js app is UI and nothing else.

Three new architectural facts.

1) Client never touches cowork-api. Every cowork-api call goes through a Next.js Route Handler. XO_API_KEY and COWORK_API_URL are server-only env (no NEXT_PUBLIC_). Per workspace CLAUDE.md rule.

2) Multi-tenant via Clerk. The phone OS wires @clerk/nextjs. Every Next proxy reads the Clerk session, extracts the username, and forwards it to cowork-api on every request. Signed-in users see their workspace; signed-out visitors fall back to the operator's workspace (a "demo" view of yours). cowork-api scopes data by username on its side.

3) cowork endpoints render as apps. Each useful cowork-api endpoint becomes a kind: "api" app tile in Agent mode: Sessions, Files, Agents, Channels, Connectors, Usage, etc. Adding a new cowork app is one app/<route>/app.ts + one Next proxy route + one renderer. Adding cowork apps pushes the Agent grid past one screen, so Agent mode supports iOS-style sub-pages within the mode.

4) Agent can drive the OS. The cowork-api chat stream extends its SSE event taxonomy with a new client-action event: {kind, ...args}. The phone OS listens on the same EventSource it already uses for text deltas and runs an action dispatcher. Kinds include navigate, go-home, pop-back, open-overlay, close-overlay, collapse-chat. User asks "show me my files" → agent emits {kind: "navigate", route: "/files"} → the OS routes there. The chat stays open while the OS reframes. This is what makes the phone feel like the agent is using it, not just answering questions in a box.

5) One chat session per user. Each user has exactly one ongoing chat session with the agent, ever. No session picker, no "new chat" button, no session history list. The chat dock button always opens the user's single conversation. session_id is resolved server-side from Clerk username on every chat call; no client-side session_id management needed. The "Sessions" cowork app drops out of the catalog (a list of one is not an app); the conversation IS the dock chat.

2.Three modes, one swipeable pager

No more mode toggle in Control Center. No mode banner. Modes are pages. Swiping between pages fires the existing mode-change choreography (~600ms): dock slides, wallpaper crossfades, status bar tints. The swipe is the trigger; the mode change is the substance. The lockscreen guards all three; on unlock the visitor lands on Landing (page 1).

iOS-style sub-pages, within a mode. When a mode's app grid overflows one screen (Agent does, with cowork-api apps added), the mode itself paginates horizontally exactly like an iPhone home screen. Page dots above the dock, snap physics, peek-on-drag. Key departure from iOS: the dock changes between modes (not between sub-pages of the same mode). Inside a mode, dock is constant across all sub-pages.

Dock convention, every mode. Leftmost slot is always Chat. Rightmost slot is auth-conditional: when signed out, it's the external Sign Up link (new tab); when signed in, it becomes Create Your Own (workspace creation flow; target TBD). The middle two slots change per mode. That convention is what makes the swipe feel coherent: visitor sees the same Chat anchor on the left and the same conversion exit on the right, only the middle moves between modes.

Chat lives in the dock. The leftmost dock button on every mode is Chat. Tap it: the button expands from its position into a full-screen chat surface. The app grid icons fade out, and the rest of the dock morphs into the chat input bar. Back chevron collapses it back to the button. No separate "chat widget" above the dock; the dock Chat button is the entry point on every page. The expanded state replaces what v0 called the /ask page.

Overlays consolidated to one. Spotlight (pull-down on grid), Control Center (top-right swipe), Notification panel (top-left swipe) collapse into one panel triggered by a single top swipe-down (anywhere along the top edge, no more corner zones). The combined panel is scrollable with sections for notifications, control toggles, and search. Renamed and unified under "Notification panel".

Per-mode auth gating. Each mode declares auth: "public" | "required". Landing = public, Agent = required, Public = public. Trying to swipe into a required mode while signed out fires the Clerk sign-in surface (Screen 12); successful sign-in completes the swipe. Pages inherit their mode's auth by default; per-page override comes later if needed. This replaces the "post-Via-demo trigger" with a deterministic rule: the gate fires exactly when the user crosses into a gated mode.
