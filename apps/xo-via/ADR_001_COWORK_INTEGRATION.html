ADR-001 Status: Proposed 2026-05-20

xo-phone-os × xo-cowork-api Integration

Architecture for how xo-phone-os connects to xo-cowork-api. All calls proxied through Next.js. Five capabilities needed. Two blockers on xo-cowork-api side.

📐Context

xo-phone-os (Next.js 16) is a phone-OS-style frontend. It needs to interact with xo-cowork-api (FastAPI, port 5002) to power projects, files, and agent chat. The workspace rule (CLAUDE.md §4) requires all API calls to be server-side only: browser never hits cowork-api directly. The phone OS needs exactly five capabilities from cowork-api:

The five capabilities needed

#CapabilityNeedCowork-api status
1 Create project Initialize a new project workspace for a user ⚠ PARTIAL — no single atomic endpoint; workaround via /api/files/mkdir?scaffold=true + POST /api/agents
2 Get project All files + metadata + agent config for a project ✗ BLOCKER — no unified endpoint; requires 2+ calls composed on Next.js side
3 Chat (one session) Send a message and stream the response. One session per user. ✓ EXISTS — two-step: POST /api/chat/prompt → GET /api/chat/stream/{id}
4 All files in a project List + read files within a project directory ✓ EXISTS — POST /api/files/list-directory + POST /api/files/content
5 Health Is cowork-api reachable? ✓ EXISTS — GET /health

⚖️Decision

Every request from the browser goes to a Next.js Route Handler first. Next.js injects auth and forwards to xo-cowork-api. The browser never sees port 5002 or any API keys.

Browser (xo-phone-os)

    │  Clerk session cookie attached automatically

🔀Options Considered

Option A (chosen): Next.js proxy layer

Browser calls /api/* on Next.js. Next.js calls cowork-api with auth injected. Keys never leave the server. One architecture rule in CLAUDE.md covers all future endpoints.

Pros: Key safety enforced architecturally. Single point for rate limiting, abuse detection, Clerk auth. CORS never an issue. Consistent with workspace rule.

Cons: Extra network hop (milliseconds on localhost, negligible). Need a Next.js route per cowork-api endpoint we expose.

Option B (rejected): Direct browser → cowork-api

Browser calls http://localhost:5002/api/* directly. XO_API_KEY would need to be prefixed NEXT_PUBLIC_, exposing it in client bundle.

Why rejected: Violates CLAUDE.md §4. Key exposed. CORS configuration required on cowork-api for every new phone-os origin. No central audit/rate-limit point.

🛣️Next.js Route Handler Design

One file per cowork-api capability. Each is ~30 LOC: read Clerk username, attach headers, forward, return. SSE routes pipe the upstream stream body directly.

  • GET /api/cowork/health → GET /health exists
  • GET /api/cowork/workspace → GET /api/config/workspace exists — returns projects root path; call this first on boot
  • POST /api/cowork/projects → Workaround: POST /api/files/mkdir (scaffold:true) + POST /api/agents partial
  • GET /api/cowork/projects/[id] → Composed: GET /api/agents/{id} + POST /api/files/list-directory blocker
  • GET /api/cowork/projects/[id]/files → POST /api/files/list-directory with {"path": project.workspace} exists
  • GET /api/cowork/projects/[id]/files/read → POST /api/files/content with {"path": filePath} exists
  • POST /api/chat → POST /api/chat/prompt — returns {stream_id, session_id} exists
  • GET /api/chat/stream/[id] → GET /api/chat/stream/{stream_id} — SSE pipe exists
  • POST /api/chat/abort → POST /api/chat/abort exists

Route handler pattern (every route, ~30 LOC)

// app/api/cowork/health/route.ts
export async function GET(req: Request) {
  const username = await getClerkUsername(req)   // reads Clerk session
  const upstream = await fetch(
    `${process.env.COWORK_API_URL}/health`,
    {
      headers: {
        "Authorization": `Bearer ${process.env.XO_API_KEY}`,
        "X-XO-Username":  username ?? "__operator__",
      },
    }
  )
  return new Response(upstream.body, {
    status: upstream.status,
    headers: { "Content-Type": upstream.headers.get("Content-Type") ?? "application/json" },
  })
}

SSE proxy pattern (chat stream route)

// app/api/chat/stream/[id]/route.ts
export async function GET(req: Request, { params }: { params: { id: string } }) {
  const username = await getClerkUsername(req)
  const upstream = await fetch(
    `${process.env.COWORK_API_URL}/api/chat/stream/${params.id}`,
    {
      headers: {
        "Authorization": `Bearer ${process.env.XO_API_KEY}`,
        "X-XO-Username":  username ?? "__operator__",
      },
    }
  )
  // Pipe the SSE stream directly to the browser
  return new Response(upstream.body, {
    status: upstream.status,
    headers: {
      "Content-Type": "text/event-stream",
      "Cache-Control": "no-cache",
      "Connection":    "keep-alive",
    },
  })
}

🔐Auth + Session Model

Header injection (all routes)

HeaderValueWhere it comes from
Authorization Bearer ${XO_API_KEY} Server-side env. Never client-readable. Never NEXT_PUBLIC_.
X-XO-Username Clerk username, or "__operator__" if signed out Next.js reads Clerk session from cookie; falls back to operator for anonymous visitors.

Single session per user

Each user has exactly one chat session, ever. The phone OS never shows a session picker or "new chat" button. When a user sends a message, the Next.js proxy includes their username in X-XO-Username; cowork-api resolves the single session tied to that user automatically. The browser does not need to track or store a session_id across restarts — the server owns session resolution.

Consequence of single-session model

The Sessions cowork-api endpoint (GET /api/sessions/) is not used. A list of one session is not a useful UI surface. The conversation is the chat dock button. The Sessions app drops from the cowork-derived app catalog.

🚧Blockers on xo-cowork-api

Two of the five required capabilities do not have clean endpoints in xo-cowork-api today. These must be built on the cowork-api side before the phone OS can go live with full project support.

Blocker 1 — Create project (no atomic endpoint)

What we need: A single call that creates a project with a given name and returns its id and path.

Current workaround: Two-step composition — POST /api/files/mkdir with scaffold:true to build the directory tree, then POST /api/agents to register the agent record. This is fragile (if step 2 fails, cleanup needed) and leaks implementation details into the phone OS proxy.

Requested endpoint:

POST /api/projects
Request:  { "name": "my-project", "template"?: "default" }
Response: { "id": "my-project", "name": "My Project",
            "path": "/Users/me/xo-projects/my-project",
            "created_at": "ISO timestamp" }

Impact if missing: Phone OS cannot create new projects until this lands. Workaround two-step is acceptable for an early build but should not ship as the permanent solution.

Blocker 2 — Get project (no unified endpoint)

What we need: A single call that returns a project's full state: name, path, file tree, and agent configuration.

Current workaround: Three-call composition on the Next.js proxy side — GET /api/agents/{id} (name + workspace path), then POST /api/files/list-directory (file tree), then GET /api/config/workspace (workspace root, once at boot). The proxy assembles and returns a unified JSON. Works but adds N round-trips per project load.

Requested endpoint:

GET /api/projects/{id}
Response: {
  "id":         "my-project",
  "name":       "My Project",
  "path":       "/Users/me/xo-projects/my-project",
  "files": {
    "dirs":  ["src", "docs"],
    "files": ["README.md", "AGENTS.md"]
  },
  "agent": {
    "name":    "my-project",
    "backend": "claude_code",
    "workspace": "/Users/me/xo-projects/my-project"
  },
  "updated_at": "ISO timestamp"
}

Impact if missing: Phone OS loads projects with 3 serial HTTP calls instead of 1. Acceptable for early builds; should be replaced before public launch to reduce latency.

🔄Consequences

What becomes easier

  • API keys are never in the browser bundle. Zero risk of key exposure in client-side JS.
  • CORS never needs to be configured on cowork-api for the phone OS origin.
  • Rate limiting and abuse detection can be applied in one place (the Next.js proxy).
  • Adding a new cowork-api capability is one new Next.js route (~30 LOC).
  • Auth model is uniform: Clerk session everywhere, no special-casing per endpoint.

What becomes harder

  • Every new cowork-api endpoint requires a corresponding Next.js route (minor but real overhead).
  • SSE piping (chat stream) adds one extra in-flight connection on the Next.js server per active chat.
  • Debugging requires two hops: browser → Next.js logs → cowork-api logs.

What we need to revisit

  • Once Blocker 1 + 2 land, the /api/cowork/projects and /api/cowork/projects/[id] Next.js routes need to be simplified from N-call composition to single-call proxies.
  • When Clerk auth is wired (Section G, v.Next+), the getClerkUsername helper expands to return full user context; downstream routes already accept it.
  • If cowork-api ever adds a projects router natively, the phone-os Next.js routes can be slimmed without UI changes.
